# AWS

LabLumen runs entirely on AWS in `us-east-1`. The diagram below shows how every service connects — from user traffic at the edge down to the database subnet.

![AWS Infrastructure Architecture](images/aws_architecture_diagram.png)

---

## Networking

### Amazon VPC
A single VPC (`lablumen-vpc`, `10.0.0.0/16`) with three subnet tiers across two availability zones:

| Subnet | CIDR | Contents |
|---|---|---|
| Public | `10.0.101.0/24` | NAT Gateway, Internet Gateway |
| Private | `10.0.1.0/24` | EKS worker nodes, Lambda VPC-attached ENI |
| DB | `10.0.201.0/24` | RDS PostgreSQL — isolated, no internet route |

### VPC Endpoints (AWS PrivateLink)
Interface endpoints keep traffic to AWS services entirely inside the VPC — no internet traversal:

`SSM` · `Secrets Manager` · `SQS` · `ECR` · `Bedrock` · `Textract` · `CloudWatch Logs`

An S3 **gateway endpoint** handles all S3 traffic from within the VPC.

---

## Compute

### Amazon EKS
Kubernetes 1.31 cluster (`lablumen-eks`) hosting all four application services and the nginx frontend container. Uses EKS Access Entries for cluster authentication.

### Karpenter
Dynamic node autoscaling. Provisions on-demand EC2 instances (`t3.medium` / `t3.large`) when pods are pending due to insufficient capacity. Automatically consolidates underutilized nodes to reduce cost.

### AWS Lambda
Serverless runtime for the AI processing pipeline (`lablumen-ai-service`). Triggered by EventBridge, VPC-attached for private RDS access. Python 3.12, 512 MB memory, 60-second timeout.

---

## Database

### Amazon RDS PostgreSQL 16.4
Primary application database on `db.t4g.micro`, single-AZ, in the isolated DB subnet. Master credentials managed automatically by Secrets Manager. Used by `appointment-service` (appointments, lab tests, patient profiles) and `report-service` (reports, vector embeddings).

### pgvector
PostgreSQL extension installed on the RDS instance. Stores 1,536-dimension vector embeddings generated by Bedrock Titan Embed, with an HNSW cosine similarity index. Powers the AI chat RAG (Retrieval-Augmented Generation) feature in `report-service`.

### Redis (in-cluster)
Not a managed AWS service. A Redis pod runs inside EKS, used exclusively by `appointment-service` for distributed slot locking (`SET NX EX`) to prevent double-booking of time slots.

---

## Storage

### Amazon S3
Two buckets:
- **Reports bucket** — private, KMS-encrypted, versioned, with S3 Event Notifications to EventBridge on every object upload. Staff upload PDFs here; patients view them via presigned URLs.
- **SAM artifacts bucket** — used by `sam deploy` to store Lambda deployment packages.

---

## AI and Machine Learning

### Amazon Bedrock
Two models used via the Bedrock API — accessed from a cross-account IAM role because the AWS organization's SCP restricts Bedrock to a separate account:

| Model | ID | Used For |
|---|---|---|
| Amazon Nova Lite | `amazon.nova-lite-v1:0` | Generates plain-English summaries of lab reports (max 600 tokens) |
| Amazon Titan Embed | `amazon.titan-embed-text-v1` | Converts report text chunks into 1,536-dimension vectors |

### AWS Textract
OCR service. The AI Lambda calls `detect_document_text` to extract raw text from uploaded PDF lab reports before summarization and embedding.

### Amazon EventBridge
Rule: `S3 Object Created` on the reports bucket triggers the AI Lambda. Decouples S3 from Lambda — no direct S3 → Lambda integration. The Lambda processes each report asynchronously after upload.

---

## Messaging

### Amazon SQS
Standard queue (`lablumen-notifications`). `appointment-service` publishes a booking confirmation event after every successful appointment creation. `notification-service` consumes the event and sends the confirmation email.

---

## Email

### Amazon SES
Transactional email. Domain identity for `rnld101.xyz` with Easy DKIM — three CNAME records created in Route 53. Sender: `no-reply@rnld101.xyz`. Used by `notification-service` for appointment confirmation emails.

---

## Authentication

### Amazon Cognito
User Pool (`lablumen-users`) handles all user authentication. SRP flow — passwords are never sent in plaintext. ID tokens carry a `cognito:groups` claim used for RBAC across every backend service:

| Group | Role |
|---|---|
| `PATIENT` | Can book appointments and view own reports |
| `LAB_STAFF` | Can manage operations queue and upload reports |
| `LAB_ADMIN` | Same as LAB_STAFF |

No separate auth service exists. Each backend service independently verifies the Cognito JWT on every request.

---

## Security and Secrets

### AWS KMS
Platform CMK (`alias/lablumen-platform`) encrypts ECR repositories, the S3 reports bucket, and Secrets Manager secrets. Annual key rotation enabled.

### AWS Secrets Manager
Stores sensitive runtime credentials: the PostgreSQL DSN (`lablumen/app/database-url`) and Grafana admin password (`lablumen/app/grafana-admin`). External Secrets Operator fetches these and injects them as Kubernetes Secrets. The Lambda fetches the DB DSN directly at cold start.

### AWS SSM Parameter Store
Stores 15 non-sensitive configuration values under `/lablumen/config/*` — Cognito pool ID, app client ID, SQS queue URL, S3 bucket name, and others. Fetched by ESO at runtime and injected into pods alongside the sensitive secrets.

---

## Container Registry

### Amazon ECR
Four private repositories — one per service. Immutable image tags prevent any tag from being overwritten after push. Scan-on-push enabled for automatic vulnerability detection. KMS-encrypted.

---

## Load Balancing and DNS

### AWS Application Load Balancer (ALB)
Single internet-facing ALB on HTTPS port 443 (`IngressGroup: lablumen`). Created and fully managed by the AWS Load Balancer Controller from Kubernetes Ingress objects. All external traffic enters here.

### AWS Certificate Manager (ACM)
Wildcard certificate (`*.rnld101.xyz`) attached to the ALB. TLS termination happens at the ALB; traffic to pods inside the cluster is HTTP.

### Amazon Route 53
DNS for `rnld101.xyz`. A records for application hostnames are created and updated automatically by ExternalDNS based on Kubernetes Ingress annotations.

---

## Identity

### AWS IAM + GitHub OIDC
GitHub Actions authenticates to AWS via OIDC — no static access keys exist anywhere. Each workflow job exchanges a GitHub-signed JWT for short-lived STS credentials scoped to a specific IAM role.

### IRSA (IAM Roles for Service Accounts)
Each pod that needs AWS access has a dedicated IAM role bound to its Kubernetes ServiceAccount. The pod receives temporary credentials via the EKS OIDC token projection — no credentials are stored in the cluster.

### AWS STS
Issues temporary credentials for GitHub OIDC sessions and for the Lambda's cross-account Bedrock `AssumeRole` call.

---

## Observability

### Amazon CloudWatch
EKS control plane logs are sent to CloudWatch log groups. Lambda execution logs and metrics are also in CloudWatch.

### Prometheus + Grafana
Deployed in-cluster in the `monitoring` namespace via the kube-prometheus-stack Helm chart. Scrapes all pod and node metrics. Grafana admin credentials are synced from Secrets Manager via ExternalSecret.

---

## Infrastructure as Code

### AWS CloudFormation
Used only by AWS SAM under the hood to manage the AI Lambda stack (`lablumen-ai`). All other infrastructure is managed by Terraform.
