# Deployment Runbook

End-to-end guide for deploying the LabLumen platform from scratch. Follow phases in order — each phase depends on the previous one completing successfully.

---

## Prerequisites

The following must exist before starting. Terraform does **not** create them.

- An AWS account with administrator access
- A registered domain with an active Route 53 **hosted zone** (e.g., `rnld101.xyz`)
- An ACM wildcard certificate (`*.rnld101.xyz`) in `us-east-1` with status **ISSUED**
- AWS CLI configured locally with admin credentials
- Terraform 1.10+ installed locally
- `kubectl`, `helm`, and the AWS CLI installed locally
- A GitHub organization with all 8 repositories created and source code pushed

---

## Phase 1 — GitHub Organization Setup

Configure secrets, variables, and environments before any pipeline run.

### 1.1 — Organization-level variables

Go to **GitHub → Organization Settings → Secrets and variables → Actions → Variables** and create:

| Variable | Value |
|---|---|
| `AWS_ACCOUNT_ID` | Your 12-digit AWS account ID |
| `SONAR_ORGANIZATION` | Your SonarCloud organization key |

### 1.2 — Organization-level secrets

Go to **GitHub → Organization Settings → Secrets and variables → Actions → Secrets** and create:

| Secret | How to obtain |
|---|---|
| `K8S_REPO_PAT` | GitHub → Settings → Developer settings → Personal access tokens → Classic. Grant `repo` scope. This token must have write access to `lablumen-k8s`. |
| `SONAR_TOKEN` | SonarCloud → My Account → Security → Generate Token |
| `SNYK_TOKEN` | Snyk → Account Settings → Auth Token |

### 1.3 — Terraform repository secrets

On the `lablumen-terraform` repository (repo-level, **not** org-level):

| Secret | How to obtain |
|---|---|
| `INFRACOST_API_KEY` | Infracost Cloud → Org Settings → API Keys → Add API key |
| `BEDROCK_CROSS_ACCOUNT_ROLE_ARN` | ARN of the cross-account IAM role for Bedrock in the Bedrock-enabled AWS account |

### 1.4 — Terraform repository environment

On `lablumen-terraform` → **Settings → Environments → New environment**:

- Name: `production`
- Add required reviewers (yourself or the infrastructure team)

This environment is the manual approval gate. `terraform apply` will not run until a reviewer approves the plan.

## Phase 2 — Terraform State Bucket Bootstrap

The S3 state bucket must be created once using local state before the main Terraform stack can run.

```bash
cd lablumen-terraform/bootstrap
terraform init
terraform apply
```

When apply completes, copy the bucket name from the output:

```bash
terraform output -raw state_bucket
```

Open `lablumen-terraform/backend.tf` and replace the bucket name placeholder with this value. Commit and push the change.

---

## Phase 3 — Terraform Apply (All Infrastructure)

This phase creates the entire AWS infrastructure stack. **It must be run locally the first time** because the OIDC pipeline roles are created by Terraform itself — they do not exist yet, so the pipeline cannot authenticate.

```bash
cd lablumen-terraform
terraform init
terraform apply
```

Review the plan carefully before confirming. Apply takes approximately 15–20 minutes; EKS cluster creation is the longest step.

When apply completes, note the outputs — you will need them in the next phases:

| Output | Used In |
|---|---|
| EKS cluster name and endpoint | Phase 5 (`aws eks update-kubeconfig`) |
| RDS endpoint | Phase 4 (building the database DSN) |
| Cognito User Pool ID and App Client ID | Verified in SSM automatically by Terraform |
| ECR registry URL | CI/CD image push |
| S3 reports bucket name | AI Lambda configuration |

## Phase 4 — Secrets Manager Population

Terraform creates empty secret shells. Populate them with real values before the application can start.

### `lablumen/app/database-url`

Retrieve the RDS master password from Secrets Manager (it was auto-generated by Terraform):

```bash
aws secretsmanager get-secret-value \
  --secret-id lablumen-rds-master-password \
  --query SecretString --output text
```

Then populate the application DSN secret:

```bash
aws secretsmanager put-secret-value \
  --secret-id lablumen/app/database-url \
  --secret-string "postgresql+asyncpg://USERNAME:PASSWORD@RDS_ENDPOINT:5432/lablumen"
```

Replace `USERNAME`, `PASSWORD`, and `RDS_ENDPOINT` with the values from the RDS master credentials and Terraform outputs.

### `lablumen/app/grafana-admin`

```bash
aws secretsmanager put-secret-value \
  --secret-id lablumen/app/grafana-admin \
  --secret-string '{"username":"admin","password":"<choose-a-strong-password>"}'
```

## Phase 5 — ArgoCD Bootstrap

With the EKS cluster running, bootstrap ArgoCD. This single script installs ArgoCD and triggers the entire platform to self-configure.

```bash
# Point kubectl at the new cluster
aws eks update-kubeconfig --region us-east-1 --name lablumen-eks

# Bootstrap ArgoCD and apply the root Application
cd lablumen-k8s
bash scripts/bootstrap-argocd.sh
```

The script installs ArgoCD via Helm and applies `bootstrap/root-app.yaml`. ArgoCD then deploys everything in sync-wave order:

- **Wave 0** — Platform add-ons: Karpenter, External Secrets Operator, AWS Load Balancer Controller, ExternalDNS, Prometheus/Grafana
- **Wave 1** — Cluster-wide config: ESO ClusterSecretStores connecting to Secrets Manager and SSM
- **Wave 2** — All application services in `lablumen-dev` and `lablumen` namespaces

Monitor progress:

```bash
kubectl get applications -n argocd
```

All applications should reach **Healthy / Synced** within 5–10 minutes. If any application shows `OutOfSync` or `Degraded`, check its events:

```bash
kubectl describe application <app-name> -n argocd
```

## Phase 6 — CI/CD First Run

With infrastructure running and ArgoCD bootstrapped, trigger the CI/CD pipeline for each service to build and push the first Docker images.

### Backend and frontend services

For each of `appointment-service`, `report-service`, `notification-service`, and `frontend`:

1. Push a commit to `main` (or merge an open PR).
2. The `service-build-push.yml` workflow runs: builds the image, runs Trivy, pushes to ECR, updates `values-dev.yaml` in `lablumen-k8s`.
3. ArgoCD detects the change and deploys to `lablumen-dev`.

**Database migrations** — Alembic runs automatically on `appointment-service` startup. The first deploy creates all tables and the pgvector extension on RDS.

### AI Lambda service

1. Push a commit to `main` on `lablumen-ai-service`.
2. The SAM pipeline runs: `sam build --use-container` → `sam deploy`.
3. Lambda is live when the CloudFormation stack reaches `UPDATE_COMPLETE`.

```bash
aws cloudformation describe-stacks \
  --stack-name lablumen-ai \
  --query 'Stacks[0].StackStatus'
```

## Phase 7 — Validation

Verify the platform is fully operational before promoting to production.

### Infrastructure checks

```bash
# All pods running
kubectl get pods -A

# All ArgoCD applications synced and healthy
kubectl get applications -n argocd

# ALB created by Load Balancer Controller
kubectl get ingress -A

# All ExternalSecrets synced (Status = Ready)
kubectl get externalsecrets -A
```

### Browser checks

1. Open `https://app.rnld101.xyz` — the login page should load.
2. Register a new patient account and complete the multi-step booking wizard.
3. Log in as a staff member and verify the operations queue shows the booking.
4. Upload a lab report PDF from the staff operations view.
5. Wait approximately 30 seconds for the AI Lambda to process the PDF.
6. Log back in as the patient, open the report, and verify:
   - The AI summary panel is populated
   - The "Chat with AI" button is active and responds to questions

## Production Promotion

Once dev is validated, promote each service to production by publishing a GitHub Release:

1. Go to the service repository → **Releases → Draft a new release**
2. Tag it with a semver version (e.g., `v1.0.0`) targeting the `main` branch
3. Publish the release
4. The `service-release.yml` workflow retags the ECR image (no rebuild) and updates `values-prod.yaml` in `lablumen-k8s`
5. ArgoCD deploys to the `lablumen` (production) namespace

Repeat for each service. There is no required release order, but release `appointment-service` first since it owns the database schema.

---

## Teardown

To destroy all infrastructure (non-reversible):

1. Delete all ECR images from each repository (ECR repos must be empty for Terraform to delete them).
2. Empty and delete all objects in the S3 reports bucket and SAM artifacts bucket (versioned buckets must have all versions deleted).
3. Trigger the `terraform-destroy.yml` workflow manually on `lablumen-terraform` — it requires typing a confirmation string before proceeding.

All ArgoCD-managed resources are removed automatically when Terraform deletes the EKS cluster.
